Back Ground

Si parte dalla problematica di un utente che si sposta da casa per raggiungere un posto di mare e trascorrere una giornata lì. Ci sono delle necessità di trovare un ombrellone e dei servizi come lettini, sedie sdraio e il parcheggio per auto o moto. Le aziende fornitrici di servizi per il mare come ombrelloni e parcheggi per posti auto hanno a loro volta la necessità di uno strumento di gestione automatizzato dei propri servizi come ombrelloni etc.

Objective

L'obiettivo del Ombrelloni d'Italia è quello di fornire un supporto all'utente che si sposta per andare a mare in tranquillità prenotando da casa ed in anticipo i servizi che lui ritiene opportuno ed avendo anche così anche la possibilità di pagare

Architecture Overview

Ombrelloni d'Italia è una Java Enterprise application accessibile dagli utenti e dai fornitori. Ciascun fornitore di servizi italiano può registrare tutti i servizi che mette a disposizione. Gli utenti potranno prenotare e pagare i servizi di cui hanno necessità. La dispobilità dei servizi da parte degli utenti sarà successivamente erogata tramite app. Sarà necessario: un DBMS in cui saranno memorizzati sia i dati dei servizi che i dati e le operazioni di prenotazione, disdetta e pagamento. un LDAP dove sarnno memoriazzati i dati degli utenti e dei fornitori un JCR dove saranno memorizzati le eventuali immagini dei servizi erogati dai fornitori un application server EE compliant dove girerà la Java Enterprise application

Non Functional Requirement

The non-functional requirements are considered:

  1. Availability of 99.99% - Team doctor application has users, Doctor and Consultant throughout North America, Europe and Asia. To accomplish this, Team Doctor application use server clustering for load balancing and failover. Team Doctor application has the front-end as stateless as possible by using granular service. Team Doctor server is always on line and constantly checking external web services to alert WS administrator with notification email if there are problems.

  2. Reliability – Instances of Team Doctor application run simultaneously on application server in the cluster. There is a real time distributed cache to keep data synchronized across the JVMs. In clustered environment, session data is replicate in real time. This is done by saving the session data in a database and then enabling replication at the database level.

  3. Performance - In the context of Team Doctor web-based application good performance involves short response time for a submit request to assistance / response. To improve performance could use database cache and we set a cluster application that use round-robin handling to ensure that application maintains stated performance goal.

  4. Scalability – Team Doctor architecture is designed to support both vertical scalability as well as horizontal scalability. The application can scale by deploying more Web containers in multiple machines to handle more number of client requests. Most of the business components are implemented as stateless session beans RequestService, ResponseService, AdviceRequest. This enhances scalability of the application requests by bean pooling technique and maintains an optimal throughput. The business logic is not tightly coupled with web tier components.

  5. Manageability, Flexibility, Extensibility - The logical separation of Team Doctor application tiers into different tiers (Presentation, Business Logic, Integration tier) allows a system to be flexible and easily maintained. When there are any changes in presentation, it will be done at presentation tier and it will not affect the other tiers. If there are any business logic changes, only EJB components business logic get changed and it will not affect other tiers and vice versa. Team Doctor application uses a component-based architecture and uses MVC architecture (JSF) with managed bean controller at the web tier. This enhances the addition of new web actions or business components, eases maintenance of the system by isolating changes in one component from affecting other components.

Security

Any consideration of clinical data raises questions concerning the safety and security of individual patient records. Maintaining confidentiality of data records is of paramount importance. The complex issue has many challenging dimensions, from what happens after the initial intake of an individual’s data to what happens in data aggregation.

The characteristics of application security that help to minimize the security threats faced by an enterprise include the following:

  • Authentication: The means by which communicating entities, such as client and server, prove to each other that they are acting on behalf of specific identities that are authorized for access. This ensures that users doctors and consultants are who they say they are.

  • Authorization, or access control: The means by which interactions with resources are limited to collections of users or programs for the purpose of enforcing integrity, confidentiality, or availability constraints. This ensures that users have permission to perform operations or access data.

  • Data integrity: The means used to prove that information has not been modified by a third party, an entity other than the source of the information. For example, web services data sent over an open network must be able to detect and discard messages that were modified after they were sent.

  • Confidentiality, or data privacy: The means used to ensure that information is made available only to users who are authorized to access it. This ensures that only authorized users Doctor and Consultant can view sensitive data.

  • Non-repudiation: The means used to prove that a user who performed some action cannot reasonably deny having done so. This ensures that transactions can be proved to have happened.

  • Quality of Service: The means used to provide better service to selected network traffic over various technologies.

  • Auditing: The means used to capture a tamper-resistant record of security-related events for the purpose of being able to evaluate the effectiveness of security policies and mechanisms. To enable this, the system maintains a record of transactions and security information.

To implement the security features listed above assumes use of the following technologies:

  • Java Authentication and Authorization Service (JAAS): JAAS is a set of APIs that enable services to authenticate and enforce access controls upon users. JAAS provides a pluggable and extensible framework for programmatic user authentication and authorization. JAAS is a core Java SE API and is an underlying technology for Java EE security mechanisms.

  • Java Cryptography Extension (JCE): JCE provides a framework and implementations for encryption, key generation and key agreement, and Message Authentication Code (MAC) algorithms. Support for encryption includes symmetric, asymmetric, block, and stream ciphers. Block ciphers operate on groups of bytes; stream ciphers operate on one byte at a time. The software also supports secure streams and sealed objects.

  • Java Secure Sockets Extension (JSSE): JSSE provides a framework and an implementation for a Java version of the Secure Sockets Layer (SSL) and Transport Layer Security (TLS) protocols and includes functionality for data encryption, server authentication, message integrity, and optional client authentication to enable secure Internet communications.

Technology

For the project JEE technologies used are for:

Web Application Technologies

JavaServer Faces 2.0 - The greatest advantages of JavaServer Faces technology is that it offers a clean separation between behavior and presentation for web applications. JavaServer Faces technology allows you to build web applications that implement the finer-grained separation of behavior and presentation that is traditionally offered by client-side UI architectures. The separation of logic from presentation also allows to focus on a single piece of the development process and provides a simple programming model to link the pieces.

Enterprise Application Technologies

Contexts and Dependency Injection for Java (Web Beans 1.0) - The most fundamental services provided by CDI are as follows:

  • Contexts: The ability to bind the lifecycle and interactions of stateful components to well-defined but extensible lifecycle contexts

  • Dependency injection: The ability to inject components into an application in a typesafe way, including the ability to choose at deployment time which implementation of a particular interface to inject

Bean Validation 1.0 - Validating input received from the user to maintain data integrity is an important part of application logic. The Bean Validation model is supported by constraints in the form of annotations placed on a field, method, or class of a JavaBeans component, such as a managed bean.

Enterprise JavaBeans 3.1 - EJB beans are specifically designed to implement the business logic of application. As such they provide services that are often needed when implementing such logic, such as transactions, injecting of the entity manager (used for JPA, the Java Persistence API) and pooling of beans.

Java Persistence 2.0 - JPA is the latest of several Java persistence specifications. The biggest benefit of JPA is that it is independent API and easily integrates with J2EE. It needs an ORM implementation to persist the java objects into DB as Hibernate.

Java Message Service API 1.1 - The JMS API enhances the Java EE platform by simplifying enterprise development, allowing loosely coupled, reliable, asynchronous interactions among Java EE components and legacy systems capable of messaging. In Team Doctor, request of assistance are put in queue as the reponse. The mode is publisher-subscriber of persistent type.

Java Transaction API (JTA) 1.1: We can have one transaction that covers both the database and the message service.

JavaMail 1.4: The JavaMail API provides a platform-independent and protocol-independent framework to build mail and messaging applications.

Top three risks and mitigation plan

Risk

Mitigation

Problems in the invocation of the external web services: rich content repository, patient record system, prescriptions database

As soon as there is a problem with the external web services, such as connection timeout or an error in the ws response will be sent a notification email to the administrators of ws server. Team Doctor users will be presented an error message but they may continue with other operations.

New file attached to request (X-ray or MRI scan), that does not exist in external system, is not available and not replicated yet in other data center when reviewing request.

User should be presented with loading panel and explicit message. The HTML 5 WebSocket API could be used to check file availability and in case of successful replication file could be displayed to end user.

More hospitals can join the program

Provided architecture allows both vertical as well as horizontal scaling. In case of experienced slowness in clustered database due to increased number of active users caching system can be introduced. The new hospitals that will become part of Team Doctor will have to meet the same WSDL for external systems.
